* Polish/Engagement Loop
  ---
  1. Zone-Based Difficulty (Biggest Win)

  Right now all enemies level 1-10 spawn anywhere with equal probability. This is the core source
   of the "random" feeling.

  Suggestion: Divide the map into concentric zones radiating from the player's starting position
  (or from cities). Spawn enemy level based on zone:

  - Inner ring (near starting city): levels 1-3
  - Middle ring: levels 4-7
  - Outer ring / mountains / marshes: levels 8-10
  - Boss lairs: fixed locations for level 11-12

  This gives the player a sense of direction — "I need to get stronger before venturing into the
  mountains." The terrain types already map nicely to this (plains = safe, mountains/marsh =
  dangerous).

  Implementation: In SpawnEnemies() in game/world/level.go, filter the rogue pool based on the
  spawn tile's distance from the nearest city or from map center, instead of the flat level <= 10
   check.

  ---
  2. Visible Enemy Indicators

  Let the player see how dangerous an enemy is before contact. Options:

  - Tint enemy sprites by tier (green/yellow/red glow or aura)
  - Show the enemy's level number floating above them
  - Use different walking sprites per tier (you already have per-rogue sprites)

  This turns encounters from "random punishment" into risk/reward decisions: "Do I engage that
  red enemy for a better ante card, or avoid it?"

  ---
  3. Progressive Spawning Over Time

  Currently SpawnEnemies draws from the same pool regardless of game state. Tie the spawn pool to
   progression markers:

  - Days elapsed: After day 10, allow level 5+. After day 25, allow level 8+.
  - Duels won: Each 3 wins shifts the minimum spawn level up by 1.
  - Amulets collected: Each amulet unlocks the next difficulty tier.

  This creates a natural difficulty ramp even without zones.

  ---
  4. Reward Scaling & Card Economy

  The engagement loop needs better pull (reasons to seek fights), not just push (enemies chasing
  you):

  - Bounty system: Higher-level enemies drop better cards or bonus gold. Right now you only get
  their ante card — add a gold bounty scaling with enemy level.
  - City demand board: Cities could post wanted rogues with specific bounties, giving the player
  a reason to hunt specific enemies.
  - Card trader in cities: Let players sell unwanted cards for gold, creating a use for weak
  cards won from easy fights.

  ---
  5. Encounter Variety (Random Encounters)

  Your RandomEncounterScr system exists but feels like it could carry more weight:

  - Positive events: Find a treasure chest (gold/card), healing spring (restore life), wandering
  merchant (buy at discount)
  - Choice events: "A traveler offers to trade a random rare for 3 of your commons" — meaningful
  risk/reward decisions
  - Lore/map reveals: Learn which rogues patrol a region, reveal city locations on minimap

  ---
  6. Retreat / Flee Mechanic

  Getting caught by a level 10 enemy when you're weak feels bad if the only options are "duel or
  bribe." Consider:

  - Flee option in DuelAnteScr — costs food instead of gold, with a % chance based on level
  difference
  - Partial loss on flee — lose some gold but keep your cards
  - This makes the bribe system feel like a premium option rather than the only escape

  ---
  7. Quest-Driven Exploration

  Quests currently feel disconnected from the core loop. Tighten them:

  - Quest enemies appear on the map with a distinct marker so you can hunt them
  - Chain quests: Completing a delivery quest unlocks a defeat quest in the destination city
  - Quest rewards that affect the world: Completing a quest could clear high-level enemies from a
   zone, or unlock a new city shop tier

  ---
  Priority Ranking

  If I were implementing these, I'd start with:

  1. Zone-based difficulty — immediately fixes the "random" feel
  2. Visible enemy indicators — gives player agency over encounters
  3. Progressive spawning — creates session-level arc
  4. Reward scaling — makes engaging strong enemies feel worth it

  These four changes together would transform the loop from "wander and hope" to "explore, assess
   risk, grow stronger, push further" — which is the classic roguelike loop that keeps players
  coming back.

Tier 1 — High-frequency, high-impact (would unlock the most cards)

5. Return/Bounce (27 unparsed) — "Return target creature to its owner's hand", "Return target creature card from your graveyard to your hand". Core blue mechanic.

7. Sacrifice effects (91 unparsed) — "Sacrifice a creature", "As an additional cost, sacrifice...". Many cards require this.

*** Combat Sub-phases
| Beginning of Combat | "At beginning of combat" triggers; players get priority               |
| Declare Attackers   | Active player declares attackers; triggers fire; players get priority |
| Declare Blockers    | Defending player declares blockers; damage assignment order set       |
| Combat Damage       | First/double strike damage (if applicable), then regular damage       |
| End of Combat       | "At end of combat" triggers; creatures removed from combat            |


1. Beginning of Combat Step: Triggers that happen "at the beginning of combat" occur here. Players get priority.
2. Declare Attackers Step: The active player declares attackers. Turn-based actions (tapping/declaring attackers) happen first, then players get priority.
3. Declare Blockers Step: The defending player declares blockers. Turn-based actions (declaring blockers) happen, then players get priority.
4. Combat Damage Step: If creatures with first/double strike exist, this happens twice; otherwise, once. Damage is assigned, then players get priority.
5. End of Combat Step: Triggered abilities that happen "at end of combat" trigger here. Players get priority before creatures are removed from combat.

## 1. **Percentage-Based Positioning**
Instead of hardcoded pixel positions, use percentages of screen dimensions:

```go
// Instead of:
buttonX := 100
buttonY := 50

// Use:
buttonX := int(float64(screenW) * 0.1)  // 10% from left
buttonY := int(float64(screenH) * 0.065) // 6.5% from top
```

## 2. **Anchor-Based Layout System**
Create an anchor system for UI elements:

```go
type Anchor int
const (
    TopLeft Anchor = iota
    TopCenter
    TopRight
    CenterLeft
    Center
    CenterRight
    BottomLeft
    BottomCenter
    BottomRight
)

type UIElement struct {
    Anchor Anchor
    OffsetX, OffsetY int  // Offset from anchor point
}

func (ui *UIElement) GetPosition(screenW, screenH int) (int, int) {
    switch ui.Anchor {
    case TopLeft:
        return ui.OffsetX, ui.OffsetY
    case TopCenter:
        return screenW/2 + ui.OffsetX, ui.OffsetY
    case Center:
        return screenW/2 + ui.OffsetX, screenH/2 + ui.OffsetY
    // ... other cases
    }
}
```

## 3. **Responsive Layout Manager**
Create a layout manager that handles different screen sizes:

```go
type LayoutManager struct {
    baseWidth, baseHeight int  // Your original 1024x768
}

func (lm *LayoutManager) ScalePosition(x, y, currentW, currentH int) (int, int) {
    scaleX := float64(currentW) / float64(lm.baseWidth)
    scaleY := float64(currentH) / float64(lm.baseHeight)

    return int(float64(x) * scaleX), int(float64(y) * scaleY)
}
```

## 4. **Constraint-Based Layout**
Define relationships between UI elements:

```go
type Constraint struct {
    Element    *UIElement
    RelativeTo *UIElement  // nil for screen edges
    Spacing    int
    Direction  Direction   // Above, Below, LeftOf, RightOf
}
```

## 5. **Grid System**
Divide the screen into a grid and position elements within grid cells:

```go
type GridLayout struct {
    Rows, Cols int
}

func (g *GridLayout) GetCellPosition(row, col, screenW, screenH int) (int, int) {
    cellW := screenW / g.Cols
    cellH := screenH / g.Rows
    return col * cellW, row * cellH
}
```

Here's an abstract pixel representation approach:

## **Virtual Pixel System**

```go
// Virtual pixel unit - always represents the same logical size
type VPixel int
type VPoint struct {
    X, Y VPixel
}
type VRect struct {
    X, Y, W, H VPixel
}

// Global viewport that handles conversion
type Viewport struct {
    baseWidth, baseHeight int     // Your reference resolution (1024x768)
    currentWidth, currentHeight int
    scale float64
}

var GlobalViewport *Viewport

func (vp *Viewport) ToScreen(v VPixel) int {
    return int(float64(v) * vp.scale)
}

func (vp *Viewport) ToScreenPoint(p VPoint) (int, int) {
    return vp.ToScreen(p.X), vp.ToScreen(p.Y)
}

func (vp *Viewport) FromScreen(pixels int) VPixel {
    return VPixel(float64(pixels) / vp.scale)
}

func (vp *Viewport) UpdateScale(newWidth, newHeight int) {
    vp.currentWidth = newWidth
    vp.currentHeight = newHeight
    vp.scale = math.Min(
        float64(newWidth)/float64(vp.baseWidth),
        float64(newHeight)/float64(vp.baseHeight),
    )
}
```

## **Usage Example**

```go
// Instead of:
buttonX := 100
buttonY := 50

// Use:
buttonPos := VPoint{X: 100, Y: 50}

// In your draw function:
screenX, screenY := GlobalViewport.ToScreenPoint(buttonPos)
drawButton(screen, screenX, screenY)
```

## **Integration with Your Button System**

```go
type Button struct {
    // ... existing fields
    Position VPoint  // Instead of X, Y int
    Size     VPoint  // Instead of separate width/height
}

func (b *Button) Draw(screen *ebiten.Image) {
    x, y := GlobalViewport.ToScreenPoint(b.Position)
    w, h := GlobalViewport.ToScreenPoint(b.Size)

    opts := &ebiten.DrawImageOptions{}
    opts.GeoM.Scale(GlobalViewport.scale, GlobalViewport.scale)
    opts.GeoM.Translate(float64(x), float64(y))
    screen.DrawImage(b.Normal, opts)
}
```

## **Shortcomings of This Approach**

### **1. Global State Dependency**
- Creates coupling between all UI code and the global viewport
- Makes testing harder (need to initialize global state)
- Can cause issues in multi-threaded scenarios

### **2. Type Safety Issues**
- Easy to accidentally mix VPixel and regular int values
- Compiler won't catch when you forget to convert
- Can lead to subtle bugs when values get passed around

### **3. Performance Overhead**
- Every pixel access requires a multiplication
- Frequent conversions can add up, especially for complex UI
- Cache misses from extra indirection

### **4. Debugging Complexity**
- Harder to debug layout issues (need to convert values mentally)
- Print statements show virtual pixels, not actual screen positions
- Tools/debuggers show actual pixels, creating disconnect

### **5. Third-Party Library Integration**
- Ebiten and other libraries expect real pixels
- Need conversion at every boundary
- Easy to forget conversions when using library functions

### **6. Precision Loss**
- Repeated conversions can accumulate floating-point errors
- Small VPixel values might round to 0 at small scales
- Can cause alignment issues with fractional scaling

## **Alternative: Context-Based Approach**

```go
type RenderContext struct {
    scale float64
    screenW, screenH int
}

func (rc *RenderContext) ScaleX(x int) int {
    return int(float64(x) * rc.scale)
}

// Pass context explicitly instead of global state
func (b *Button) Draw(screen *ebiten.Image, ctx *RenderContext) {
    x := ctx.ScaleX(b.X)
    y := ctx.ScaleY(b.Y)
    // ...
}
```

This avoids global state but requires passing the context around, which you wanted to avoid.

## **Recommendation**

For your game, I'd suggest a **hybrid approach**:
- Use the virtual pixel system for **UI layout and positioning**
- Keep the global viewport for convenience
- Add helper methods to your existing structs to handle conversion
- Use real pixels for performance-critical rendering code

This gives you the convenience without the full overhead everywhere.
